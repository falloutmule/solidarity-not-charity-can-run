SNC Can Run — Customizable Control Scheme

Design target: portrait-first mobile controls for Solidarity Not Charity Can Run, using the current project direction: stacked phone layout, gameplay window on top, control deck below, MOVE bottom-left, minimap bottom-center, right-side LOOK/actions, MENU-as-pause on phone, and an editable control profile system.

portrait-first customizable safe-area aware pointer events no fragile hold-sprint Hermes/Codex-safe

1. Core recommendation

The control scheme should keep the SNC identity already emerging in the project instead of switching to a generic dual-stick shooter layout.

Default layout

Top: landscape-style 3D play window. Bottom: dedicated control deck. This prevents the old “black middle” problem where controls visually existed but felt detached from play.

MOVE stays bottom-left

The left thumb owns movement only. It should never share space with minimap, pause/menu, sprint, or give.

LOOK becomes right-side strip

LOOK should be a thumb strip / aim zone on the right, not tiny arrows. It can support tap, drag, and hold-turn behavior.

Actions are buttons, not gestures

GIVE / USE and SPRINT need large explicit controls. Sprint should be burst/toggle, not a fragile hold.

4. Customization model

Customization should be declarative. The game reads a saved profile, lays out controls from data, and writes changes back through the Options → Edit Controls flow.

5. Edit Controls flow

  1. Player opens MENU → OPTIONS → EDIT CONTROLS.
  2. Options overlay hides or dims so it does not block the controls being edited.
  3. Each control shows its boundary, label, resize handles, and reset button.
  4. Player drags controls inside the legal control deck / map band.
  5. Player uses + and − to resize selected control.
  6. Player taps Save, Reset Portrait, Reset All, or Export Profile.

The editor must clamp controls away from browser gesture edges and safe-area cutouts. It should never allow a critical button to sit under Android navigation, iOS home indicator, a notch, or the browser back gesture edge.

6. Runtime input rules

Pointer ownership

Each active pointer gets one owner: move, look, give, sprint, menu, map, or UI. Do not let one pointer trigger two gameplay controls.

Pointer capture

When a pointer starts on MOVE or LOOK, capture it until pointerup or pointercancel. Dragging off the visible circle should still work.

Cancel safety

On pointercancel, blur, orientation change, pause, or scene transition, zero movement and look input immediately.

Action sampling

DOM events update raw pointer/key state only. The game loop samples the action map once per frame.

Input pipeline

Pointer/keyboard/gamepad events
        ↓
INPUT module stores raw state by pointerId/key
        ↓
CONTROL_LAYOUT assigns raw input to control zones
        ↓
ACTION_MAP emits move/look/give/sprint/menu/map
        ↓
GAME UPDATE consumes actions
        ↓
RENDER draws world and UI feedback

7. Implementation notes for SNC

  • Keep controls declarative. Do not hardcode button coordinates in five places.
  • Keep MENU as the phone pause path. A separate tiny PAUSE button is redundant if MENU opens the pause/options overlay.
  • Keep minimap independent. Minimap should live in its own bottom-center tab/band, not inside MOVE or LOOK zones.
  • Keep sprint forgiving. Recommended default: tap for a short burst, double-tap for toggle, stamina drains normally.
  • Keep LOOK forgiving. Treat a broad right strip as the input area; visible arrows can be decoration, not the only hit targets.
  • Keep safe-area math centralized. Resize/orientation changes should rebuild the layout from the profile.
  • Do not couple visual size to gameplay strength. Making SPRINT bigger should not change sprint speed.

11. Final design contract

SNC controls should feel like a fixed phone gamepad attached to a raycaster window. The playfield remains readable. The control deck remains touchable. The minimap has its own stable home. MOVE, LOOK, GIVE, SPRINT, and MENU are all customizable but still bound by a layout contract that protects the game from regression.
