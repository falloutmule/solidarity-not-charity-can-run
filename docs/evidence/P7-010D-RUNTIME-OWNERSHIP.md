# P7-010D — Runtime ownership audit

**Scope:** read-only ownership map for `main` baseline `aef4489` / `chromeinput2`.
**Result:** PASS — runtime seams are mapped. No source, test, metadata, build, or release action was changed.

Risk notation: **L1/L2/L3** = local, cross-file, or user-visible/high-impact change surface; **R** = likely regression vector; **U** = remaining behavior uncertainty.

## Initialization and frame order

`src/build-manifest.json` is executable ordering authority. It loads canvas/resolution (`game-01`), fullscreen/viewport (`game-02`), state (`game-05`), mobile input/menu/init (`game-06`–`08`), world/runtime modules, audio (`game-13`), persistence (`game-14`), renderer/minimap/HUD/actions/input/overlays, then the loop (`game-22`).

Important initialization edges:

- `game-01` obtains `#view` and creates the internal `buf`/`zbuffer`; `game-02` immediately calls `resize()` and installs window, fullscreen, orientation, and `visualViewport` listeners.
- `game-06` declares mobile state, layout/cache/override owners and listeners; `game-08` immediately binds input and onboarding handlers. The handlers may safely reference `options` later, but options do not exist until `game-14`.
- `game-13` binds the AudioContext gesture gate before options load. It does not create an AudioContext until a gesture or sound request. `game-14` then loads profile/options/stats/leaderboards, applies URL flags, migrates mobile geometry, applies control styles, and selects mobile mode.
- `game-22` starts the sole `requestAnimationFrame` loop. Each frame does input action consumption, fixed-step simulation, then rendering: `drawScene` to `buf`, `buf` to `#view`, canvas HUD/minimap/overlays, HTML mobile-menu reconciliation, and optional query-gated perf overlay. Rendering should remain read-only with respect to game/save state.

## Reachability and safe extraction candidates

| Candidate | Reachability / constraint | Classification |
|---|---|---|
| `crResolveRenderProfile`, `crCanonicalRenderProfile`, `crRenderTargetsMatch`, and profile-proof object construction (`game-01a`) | Startup/profile application reaches them; they have no save, audio, DOM-menu, or gameplay writes. Keep target allocation/reset in the runtime owner. | **L1, R low, U low.** Best pure-calculation boundary. Existing `test:farfield-resolution`. |
| `playfieldLayout`, `portraitPlayH`, rectangle overlap/shift helpers (`game-22`) | Called by the loop/compositing and portrait layout. They are calculation-oriented but presently read globals (`view`, `RW/RH`, `mobileMode`, `options`). Extract only as dependency-injected pure helpers; do not move DOM/control application. | **L1/L2, R medium, U low.** Existing resolution/mobile cache tests cover outputs indirectly. |
| HUD formatting/debug helpers (`staminaBar`, `upgradeTag`, `crFmtFacadeDebugValue`, `crWrapDebugText`, `crFacadeDebugLines`) | Canvas-only consumers under `drawHUD`; no save/DOM/audio writes. Preserve draw order and text output. | **L1, R low/medium, U low.** Candidate for drawing-helper consolidation, not a visual redesign. |
| Portrait minimap calculation (`computePortraitMinimapDraw`) | Canvas-only path under `drawMinap`, but reads mobile/orientation/options and must stay coordinate-compatible with `portraitLayout`. | **L1/L2, R medium, U low.** Prefer a pure input/output helper with the existing caller retained. |
| Mobile UI/layout key construction (`crBuildMobileUiKey`, `crBuildMobileLayoutKey`) | Hot-path gating for `drawMobileMenu`; depends on state/options/viewport/cache revisions. | **L2, R medium/high, U medium.** Extracting can silently defeat stable early-outs; retain boundary unless profiling selects it. |
| HUD group split or viewport abstraction | Reachable every gameplay frame and coupled by shared canvas context/order. | **L2, R medium, U medium.** Only move read-only calculation/drawing helpers; preserve the current frame owner in `game-22`. |
| Save, options schema, audio unlock/cue lifecycle, control override persistence | All are lifecycle/storage/browser-policy owners with direct user data or device behavior. | **L3, R high, U medium.** Explicitly out of scope for cleanup extraction; do not combine with rendering/layout work. |

## Recommended constraint

For a bounded cleanup, select one **L1** calculation or canvas-only drawing helper and retain the caller, state owner, DOM writes, storage keys, and initialization order unchanged. Do not alter `SAVE`, `options`, control overrides, AudioContext gate/mute behavior, or audio cue timing without a separately scoped behavior card and focused verification.
